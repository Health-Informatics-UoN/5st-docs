There are a few ways to download the deployment files.

<Tabs items={["HTTPS Clone", "SSH Clone", "GitHub CLI", "Download Zip"]}>

    ```
    
  </Tabs.Tab>
  <Tabs.Tab title="SSH Clone">

    git clone git@github.com:SwanseaUniversityMedical/5S-TES-Deployment.git
    ```
    
  </Tabs.Tab>
  <Tabs.Tab title="GitHub CLI">
    Clone using the GitHub CLI.
    
    ```bash copy
    gh repo clone SwanseaUniversityMedical/5S-TES-Deployment
    ```
    
  </Tabs.Tab>
  <Tabs.Tab title="Download Zip">

    - Then click on the "Download ZIP" button.
    - Save the file to your local machine.
    - Unzip the file.
    - The repository will be in the `5S-TES-Deployment` directory.
   
   - Alternatively, run the command:
    ```bash copy

    ```
    
  </Tabs.Tab>

## Prepare the necessary files

If you would like to modify the services while keeping the original files structure (to make Git tracking easier, for example), you can follow the steps below.
- Create a new `compose.yml` file next to `DeploymentStack` and `ServiceStack` directories.

<FileTree>
    <FileTree.File name="compose.yml" />
    <FileTree.Folder name="DeploymentStack" />
    <FileTree.Folder name="ServiceStack" />
</FileTree>

- Add the following content to the `compose.yml` file, to make the connection with the original TRE's docker-compose.yml file:

```yaml copy
include:
    - DeploymentStack/TRE/docker-compose.yml
```

- Add the override settings to the `compose.yml` file if there are any. For example, if you want to change the port of the TRE Agent UI to `8123`, the content of `compose.yml` should be:

```yaml copy
include:
    - DeploymentStack/TRE/docker-compose.yml

services: 
  tre-ui:
    ports: !override # To override the port of the TRE Agent UI, to 8123 in this example
      - 8123:8080
```


If you would like to change the directory structure (e.g., to have your own way of managing the files), there are some notes below.
- We recommend to move the following files and directories to a separate directory, e.g., `/srv/tre`:

    - `DeploymentStack/TRE/docker-compose.yml`
    - `DeploymentStack/TRE/config/`
    - `ServiceStack/`

The `docker-compose.yml` file in the `DeploymentStack/TRE/` directory includes the services for the TRE Agent and Data Egress using relative paths to the compose manifests in the `ServiceStack/` directory, where they are defined.
The services inside the `ServiceStack/` directory are using the configuration files in the `config/` directory via `CONFIG_PATH` environment variable.

<Callout type="info">
`CONFIG_PATH` is the relative path from the services (yaml files) inside the `ServiceStack/` directory to the configuration files (which is under `config/` directory).
For example, the `CONFIG_PATH` for the scenario below is `../../../config` because `platform.yml` is nested 3 levels deep from the `ServiceStack/` directory compared to the `config/` directory.

<FileTree>
    <FileTree.File name="docker-compose.yml" />
    <FileTree.Folder name="ServiceStack" >
        <FileTree.Folder name="compose-manifests" >
            <FileTree.Folder name="application" >
                <FileTree.File name="platform.yml" />
            </FileTree.Folder>
        </FileTree.Folder>
    </FileTree.Folder>
    <FileTree.Folder name="config">
        <FileTree.File name="config.json" />
    </FileTree.Folder>
</FileTree>

</Callout>

- Since the directory structure is changed, you need to update the relative paths in the `docker-compose.yml` and `CONFIG_PATH` in the `.env` file accordingly.

__3.3 Set up Vault for secret management__

You need to setup Vault and generate a root token. Run the following commands one by one in the terminal to setup Vault using Docker interaction mode:

```bash
     docker compose up -d vault   # start only the Vault server
     docker exec -it vault vault operator init -key-shares=1 -key-threshold=1  # this generates and shows on the console the unseal key and the initial root token
     docker exec -it vault vault operator unseal <UNSEAL_KEY>  # unseal the vault
     docker exec -it vault vault login <INITIAL_ROOT_TOKEN> # login to the vault
     docker exec -it vault vault secrets enable -path=secret kv-v2 # enable the secret engine
```

Copy the generated root token and paste it into the `VaultRootToken` environment variable of the `.env` file. Also, __save the unseal key__ in a safe place to unseal the vault in the production mode.

<Callout type="info">
  After you restart the services using `docker compose up -d`, you may see the `vault` service keeps waiting. This is because the vault is not unsealed. 
  To unseal the vault: On another terminal, run the following command: `docker exec -it vault vault operator unseal <UNSEAL_KEY>`.
  After a few seconds, the `vault` service should start and other services will start as well.
</Callout>

__3.5 Configure KeyCloak__

Depending on where and how you want to deploy KeyCloak, the configuration is different.

<Tabs items={["Built-in KeyCloak", "Remote self-hosted KeyCloak"]}>
  <Tabs.Tab title="Built-in KeyCloak">
  `Built-in KeyCloak` means the KeyCloak service defined in the `ServiceStack/compose-manifests/shared/auth.yml` directory, and this ervice will be deployed together with the TRE services if you use the default configuration.

  If so, you need to configure the KeyCloak server by the following environment variables in the `.env` file.

<Table>
  <thead>
    <Table.Tr>
      <Table.Th>Environment Variable</Table.Th>
      <Table.Th>Description</Table.Th>
    </Table.Tr>
  </thead>
  <tbody>
    <Table.Tr>
      <Table.Td>`KeycloakBootstrapAdminPassword`</Table.Td>
      <Table.Td>
        The bootstrap admin password for the Keycloak server used by the TRE Agent. This is used to bootstrap the Keycloak server with the admin user.
      </Table.Td>
    </Table.Tr>
    <Table.Tr>
      <Table.Td>`KeycloakStartupCommand`</Table.Td>
      <Table.Td>
        The command to start the Keycloak server. In dev mode (or Demostack), this is `start-dev`.
        In production, if you have HTTPS configured for Keycloak, this should be set to `start`.
      </Table.Td>
    </Table.Tr>
    <Table.Tr>
      <Table.Td>`KeycloakFullURL`</Table.Td>
      <Table.Td>
        The full hostname URL of the Keycloak server, for example,`http://localhost:8085` or `https://my-keycloak.net`. 
        This is only used if you are using a local/internal Keycloak service defined in the `ServiceStack/compose-manifests/shared/auth.yml` file.
      </Table.Td>
    </Table.Tr>
  </tbody>
</Table>

After that, you need to start up KeyCloak server by running the following command. Necessary realms and clients configurations will be imported automatically on startup.

```bash
docker compose up -d keycloak
```

<Callout type="info">
  To access Keycloak you need to access it via the hostname of your VM and the port. You can customise these by changing the above config and refer to the default values in the TRE Components section.
</Callout>

  </Tabs.Tab>
  <Tabs.Tab title="Remote self-hosted KeyCloak">
    If you already have a KeyCloak server running remotely, TRE can still use this. However, it's recommended to import the realms and clients configurations from the `DeploymentStack/TRE/config/realm-config/` directory.
    To import the realms and clients configurations, you can follow the steps below:
    - In Keycloak Admin Console, go to the `Manage Realms` section.
    - Click on the `Create Realm` button.
    - Import the `tre-layer.json` from the `DeploymentStack/TRE/config/realm-config/` directory as `Resource File`.
    - Input the Realm name as `Dare-TRE`.
    - Keep the `Enabled` checkbox checked.
    - Click on the `Create` button.
    - Repeat the same process for the `egress-layer.json` file to create the `Data-Egress` realm.


The last step is using the URL of your KeyCloak server in the `.env` file as `TRE_KEYCLOAK_URL`.

  </Tabs.Tab>
